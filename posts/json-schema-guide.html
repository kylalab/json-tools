JSON 스키마 & 유효성 검사 완전 가이드

2025-10-01 · 작성자 Kyla

JSON은 단순하지만, 데이터가 커지고 팀 간 교환이 많아질수록 형식 보장이 필요합니다. 이때 쓰이는 것이 바로 JSON Schema입니다. 이번 글에서는 스키마의 필요성, draft-07 주요 키워드, 실무 예시, 브라우저 검증 방법, 그리고 Quick JSON의 Validate 기능과의 역할 분담을 다룹니다.

왜 JSON 스키마가 필요한가?

  • 프론트엔드 — 서버에 보내기 전 입력값 검증
  • 백엔드 — API 요청/응답이 계약(contract)에 맞는지 보장
  • 데이터팀 — 로그/이벤트 스키마를 문서화하고 품질 관리

스키마 없이는 팀마다 데이터 해석이 달라지고, 디버깅 비용이 커집니다.

JSON Schema (draft-07) 주요 키워드

{
  "type": "object",
  "properties": {
    "id": { "type": "integer" },
    "name": { "type": "string" },
    "email": { "type": "string", "pattern": "^[^@]+@[^@]+$" },
    "role": { "enum": ["user", "admin"] }
  },
  "required": ["id", "name", "email"]
}
  • type — 데이터 유형 지정
  • properties — 객체 필드 정의
  • required — 반드시 포함해야 하는 필드
  • enum — 허용 가능한 값 집합
  • pattern — 정규식 패턴 매칭

실무 예시

회원가입 요청

{
  "type": "object",
  "properties": {
    "username": { "type": "string", "minLength": 3 },
    "password": { "type": "string", "minLength": 8 },
    "age": { "type": "integer", "minimum": 13 }
  },
  "required": ["username", "password"]
}

결제 요청

{
  "type": "object",
  "properties": {
    "orderId": { "type": "string" },
    "amount": { "type": "number", "minimum": 0 },
    "currency": { "enum": ["USD", "KRW", "JPY"] }
  },
  "required": ["orderId", "amount", "currency"]
}

브라우저에서 가벼운 검증

  • 라이브러리 없이 — 간단한 경우 정규식/타입 체크만으로도 충분
  • 라이브러리 활용 — ajv 같은 검증기 사용 시 draft-07까지 폭넓게 지원

작은 프로젝트 → 가벼운 자체 검사. 큰 프로젝트 → 스키마 기반 자동 검증 추천.

Quick JSON의 Validate와의 차이

  • Quick JSON — JSON 문법 오류 위치를 빠르게 찾는 용도
  • JSON Schema — 문법이 맞더라도 “의도된 구조”를 따르는지 검증

따라서 두 기능은 경쟁이 아니라 보완 관계입니다.

체크리스트

  • 반드시 필요한 필드는 required에 추가했는가?
  • 숫자 범위, 문자열 길이 제한을 정의했는가?
  • 이메일/전화번호 등은 pattern으로 검증했는가?
  • enum 값 목록은 최신 상태인가?
